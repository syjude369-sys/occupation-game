# Occupation Game — Furniture Engine Ruleset

## 0. Purpose

This document defines the first implementation rules for the `Furniture Engine`.

The engine does **not** assign one fixed program to a motif.

Instead, it searches for a **mixed furniture layout** containing as many useful and varied furniture clusters as possible while preserving a continuous path to every required access side.

The engine is an **approximate layout optimizer**:

```text
generate many legal placements
→ combine them into mixed layouts
→ reject inaccessible layouts
→ compare valid layouts
→ preserve the highest-ranked alternatives
```

The first implementation should remain simple, debuggable, and deterministic under the same catalog, motif, settings, and seed.

---

# 1. Fixed Spatial Conventions

## 1.1 Main grid

```text
1 motif cell = 1500 mm × 1500 mm
```

The 1500 mm grid is frozen.

## 1.2 Fine grid

Furniture geometry, residual strips, access space, and circulation are evaluated on:

```text
fineGridMm = 100
```

This value should remain configurable, but `100 mm` is the initial default.

## 1.3 Motif input

The engine receives the raw motif without modifying it.

Minimum structure:

```js
{
  motifId,
  occupiedMask,
  emptyMask,
  width,
  height,
  cellSizeMm
}
```

Interpretation:

```text
O cell
= primary furniture placement territory

E cell
= existing empty space inside the motif
= circulation / buffer / planting / court candidate
```

Unused portions of an O cell may also become circulation or planting after furniture placement.

The engine must never absorb exterior cells into the motif.

# 3. Two Placement Classes

Every asset belongs to one of two geometric placement classes.

## 3.1 Grid-filling assets

Examples:

```text
conference_10
meeting_table_06
focus_pod_2p
focus_pod_1p
lounge_sofa_group
office_cubicle
plant_indoor_large
```

Rules:

```text
anchor position = 1500 mm grid intersection
footprint = exact integer-cell planning rectangle
no arbitrary offset inside the occupied cells
```

These assets snap directly to the motif grid.

The Rhino/CAD block is inserted using its predefined local base point inside the planning footprint.

For centered assets, the default block insertion is the planning-footprint center.

For directional assets such as focus pods, the block transform must preserve the door/access-side metadata.

## 3.2 Partial-cell assets

Examples:

```text
workstation_single
credenza_150
wall_storage_180
task_chair
stool_side
```

These assets are not centered automatically.

Candidate placements are generated by pushing the asset toward one side or corner of the available host region.

Initial alignment candidates:

```text
top
bottom
left
right
top-left
top-right
bottom-left
bottom-right
```

The purpose is to consolidate the solid object on one side and preserve one continuous residual space rather than two narrow fragments.

---

# 4. Allowed Transformations

Arbitrary angles are not allowed.

Allowed transformations:

```text
rotation = 0 / 90 / 180 / 270 degrees
mirror = false / true
```

Mirror means one local left-right reflection.

Rotation plus mirror is sufficient to produce all required orthogonal reflected states.

Symmetric duplicates should be removed automatically.

For asymmetric assets, especially assets with a door or explicit access side, mirrored candidates remain distinct.

---

# 5. Candidate Placement Generation

## 5.1 Grid-filling candidate generation

For each asset:

```text
for every valid grid anchor
for every allowed rotation
for mirror false / true
generate candidate
```

Reject the candidate immediately when:

```text
planning footprint leaves the motif
planning footprint uses an E cell
planning footprint overlaps an existing hard footprint
transformation is a geometric duplicate
```

## 5.2 Partial-cell candidate generation

Partial-cell assets are evaluated on the 100 mm fine grid.

For each compatible O-cell region:

```text
place asset flush to one side or corner
calculate the residual free strip
preserve the largest continuous residual region
```

Do not center a partial-cell asset unless no edge-aligned candidate exists.

Example:

```text
1500 × 1500 host cell
workstation = 1500 × 900

workstation pushed to one side
→ one continuous 600 mm residual strip
```

This residual strip is classified later as circulation, planting, buffer, or unused space.

# 6. Free Space and Opening Discovery

The motif does not contain a predefined entrance edge.

An opening is discovered from the final free-space geometry.

## 6.1 Free-space raster

After placing furniture:

```text
free space
= all motif interior fine-grid pixels
- furniture hard footprints
```

Required access/clearance zones remain free space for walking, but other hard furniture may not overlap them.

## 6.2 Exterior-connected free space

Divide the remaining free space into connected components.

A free-space component is considered externally open when it touches any part of the motif boundary.

```text
boundary-touching free component
= opening network
= possible entrance and circulation network
```

No specific side is selected in advance.

Any side is acceptable as long as a continuous free path reaches the motif boundary.

Multiple boundary-touching components may exist, but every required furniture access zone must connect to at least one of them.

---

# 7. Access Rules

Access is the primary hard constraint.

## 7.1 Minimum circulation width

Initial defaults:

```text
minPathWidthMm = 900
preferredPathWidthMm = 1500
```

A path narrower than `900 mm` is not valid circulation.

The preferred width is only a quality bonus.

## 7.2 Required access side

Each primary furniture item has one or more valid access sides.

Examples:

```text
focus pod
→ door side

workstation
→ chair side

credenza / storage
→ front side

meeting / conference / lounge
→ at least one valid side
```

A placement is accessible only when:

```text
required access zone
→ connects to a boundary-touching free-space component
→ through a route with valid minimum width
```

## 7.3 Hard validity gate

A complete layout is valid only when:

```text
1. every hard footprint stays inside its legal motif territory
2. hard footprints do not overlap
3. required clearance zones contain no other hard footprint
4. every primary furniture access zone reaches an exterior-connected path
5. every required route satisfies the minimum width
6. planting does not block a required route
```

If any rule fails:

```text
layout.valid = false
```

Invalid layouts are never allowed to win through a high score elsewhere.

---

# 8. Residual Space Classification

Residual space is classified after primary furniture placement.

## 8.1 Valid path

When a residual strip or region has:

```text
clear width >= minPathWidthMm
```

and belongs to an exterior-connected component:

```text
classification = path
```

## 8.2 Small planting pocket

When a residual strip is too narrow for circulation:

```text
clear width < minPathWidthMm
```

but large enough to hold a small planting element:

```text
classification = small_planting
```

Initial provisional parameter:

```text
smallPlantMinWidthMm = 300
```

This value must be configurable.

The first implementation may return a planting zone even when a dedicated small-plant CAD block has not yet been loaded.

## 8.3 Other residual space

If the space supports neither valid circulation nor planting:

```text
classification = buffer | unused
```

## 8.4 Large planting

`plant_indoor_large` uses a full 1500 × 1500 region.

It may be placed only after all required paths have been preserved.

Large planting never replaces required access space.

---

# 9. Mixed Furniture Layout

The engine does not generate separate program scenarios.

A single layout may contain a mixture such as:

```text
meeting_table_06 × 1
focus_pod_1p × 2
workstation set × 3
lounge_sofa_group × 1
credenza_150 × 1
plant_indoor_large × 1
```

The engine should encourage different cluster types, but access and larger furniture placement take priority.

## 9.1 Furniture clusters

Examples:

```text
conference cluster
meeting cluster
focus cluster
cubicle cluster
workstation cluster
lounge cluster
storage support
planting
```

`task_chair` does not count as a separate cluster when paired with a workstation.

Initial workstation cluster:

```text
workstation_single
+ task_chair
= workstation_set_01
```

The workstation set capacity remains `1`, not `2`.

`stool_side` has zero capacity contribution unless a future furniture-set definition explicitly assigns it a seat.

# 10. Large-Furniture Priority

The intended behavior is:

> When a larger valid furniture item fits without breaking access, prefer it over a collection of smaller items.

Do not rely only on a normal weighted sum, because many small items may otherwise outscore one large item.

## 10.1 Primary comparison key

For each valid layout:

1. collect the planning area of every placed primary cluster
2. sort those areas from largest to smallest
3. compare the resulting lists lexicographically

Example:

```text
Layout A areas = [12, 2, 1] cells
Layout B areas = [6, 6, 6] cells

Layout A wins because 12 > 6 at the first comparison
```

This directly enforces the large-furniture-first intention.

Partial-cell furniture uses actual planning area:

```text
workstation_single = 0.6 cell
credenza_150 = 0.3 cell
```

## 10.2 Secondary comparison criteria

When two layouts have the same large-furniture sequence, compare:

```text
1. cluster type diversity
2. total capacity
3. opening-side free-space quality
4. total useful occupied area
5. path quality
6. residual-space quality
```

---

# 11. Opening-Side Free-Space Quality

Furniture receives a higher score when its access side opens toward a larger continuous free area.

For every primary furniture item:

```text
measure free area directly connected to its access side
measure whether that free area reaches the motif boundary
measure the usable width of the connection
```

A good placement has:

```text
clear free area in front of the access side
no immediate obstruction
continuous connection to a boundary-opening network
```

This is a soft score because basic accessibility is already a hard requirement.

---

# 12. Optimization Method

## 12.1 Recommended first solver

Use:

```text
beam search
```

This is a heuristic optimizer.

It searches many alternative layouts but does not guarantee the mathematical global optimum unless the beam is unlimited.

The first version should describe results as:

```text
best layouts found under the configured search budget
```

## 12.2 Search order

Start from an empty motif.

Sort furniture candidates by descending planning area.

At each step:

```text
1. take every currently retained layout
2. add one legal furniture candidate
3. also preserve a skip branch
4. reject collisions and inaccessible states
5. rank remaining states
6. retain only the best K states
7. repeat until no useful insertion remains
```

## 12.3 Why access can be checked after each insertion

Adding furniture can only remove free space.

Therefore, if an access route is already impossible after an insertion, adding more furniture cannot restore it.

Such a state may be pruned immediately.

## 12.4 Initial parameters

```text
beamWidth = 50
maxReturnedLayouts = 5
maxSearchDepth = derived from motif area
```

All values must remain configurable.

## 12.5 Repeatable assets

An asset may repeat while legal space remains.

Optional metadata:

```js
{
  "repeatable": true,
  "maxCopies": null
}
```

When `maxCopies` is null, the geometric area of the motif provides the natural upper bound.

# 13. Layout Ranking

Each valid layout produces a structured ranking key.

Recommended structure:

```js
{
  valid: true,

  primarySizeSequence: [12, 2, 1, 0.6],

  diversityScore: 4,
  capacity: 16,

  openingSpaceScore: 0.82,
  occupiedAreaRatio: 0.71,
  pathQualityScore: 0.76,
  residualQualityScore: 0.63
}
```

Comparison order:

```text
A. valid layout over invalid layout
B. primarySizeSequence, lexicographic descending
C. diversityScore descending
D. capacity descending
E. openingSpaceScore descending
F. occupiedAreaRatio descending
G. pathQualityScore descending
H. residualQualityScore descending
```

This is intentionally not a single weighted number in the first implementation.

A final display score may be calculated for UI purposes, but solver ranking should use the ordered criteria above.

---

# 14. Output Alternatives

Return several non-duplicate layouts.

Suggested labels:

```text
Best Overall
Spatial Alternative 01
Spatial Alternative 02
More Diverse
Higher Capacity
```

These are all produced by the same mixed-layout optimizer.

They are not separate program scenarios.

## 14.1 Duplicate removal

Two layouts are duplicates when they contain nearly the same:

```text
asset IDs
approximate positions
rotations
mirror states
```

Initial similarity threshold:

```text
duplicateThreshold = 0.75
```

Keep the higher-ranked result.

# 16. Implementation Order

Implement in this order.

```text
1. load furniture catalog
2. generate transforms
3. generate snapped grid-filling candidates
4. generate edge-aligned partial-cell candidates
5. rasterize placed hard footprints on the 100 mm grid
6. discover boundary-touching free-space components
7. validate furniture access zones
8. classify residual strips as path / planting / buffer
9. implement large-furniture-first ranking
10. implement beam search
11. remove duplicate results
12. render top alternatives
```

Do not implement wall constraints before the above sequence works.

---

# 17. Deferred Features

The following are intentionally deferred:

```text
active wall requirements
generated internal partitions
arbitrary-angle rotation
external cells absorbed into motif
LLM-controlled exact coordinates
full regulatory accessibility verification
precise small-plant CAD asset
multi-floor routing
```

Wall metadata should remain available for later activation.

---

# 18. Core Algorithm Summary

```text
snap large/full-cell furniture to the 1500 mm grid
→ push partial-cell furniture to one side or corner
→ calculate the remaining free space
→ treat any sufficiently wide boundary-connected free space as circulation
→ treat narrow residual pockets as small planting when possible
→ reject layouts where any primary access side cannot reach the boundary
→ compare only valid layouts
→ prioritize layouts containing the largest furniture
→ use diversity, capacity, opening space, and path quality as later tie-breakers
→ preserve several distinct top layouts
```

This is the authoritative first-version rule set unless a later document explicitly replaces it.
